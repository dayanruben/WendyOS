WendyOS · device mesh

Your edge devices, on a first-name basis.

A container on one WendyOS device reaches a service on another by name — device-215.cloud.wendy.dev:8080 — with one line of config. It routes LAN-direct when the peer is on the same network, and falls back to the cloud relay when it isn't. No IPs to wire, no ports to forward.

How the call routes

Name → VIP → the shortest path that works

The agent is the whole data plane. It answers the mesh name inside the container, intercepts the connection, and carries it to the peer over the fastest route available.

RESOLVE

The name becomes a VIP

An agent-run resolver answers device-N.cloud.wendy.dev inside the container with a deterministic virtual IP — no lookup service, pure arithmetic.

LAN-DIRECT

Straight to the peer

The peer is found on the local network by its cloud ID and dialed over mutual-TLS with the device's own asset certificate. Lowest latency, no cloud hop.

FALLBACK

Relay when off-LAN

Not on the same network? The same connection tunnels through the cloud broker instead — same address, same code, it just works from anywhere.

The addressing rule

One entitlement. Every device gets an address.

# cloud device ID → mesh VIP
device N  →  10.99.<N÷256>.<N mod 256>

# e.g.
device 215  →  10.99.0.215
device 4097 →  10.99.16.1

Add the mesh entitlement to a service in wendy.json and it can reach the whole fleet by name:

{ "type": "network",
  "mode": "mesh",
  "serviceCIDR": "10.99.0.0/16" }
