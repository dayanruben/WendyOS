## Frequently Asked Questions

Find answers to common questions about WendyOS.

### General

<Accordions type="single">
  <Accordion title="What is Wendy all about?">
    Wendy represents a critical step in the vision of creating a unified framework and platform for edge development. Our mission is to become the **"iOS for all edge projects"**—defining the standard for edge computing just as iOS revolutionized mobile development.

    **The Problem We're Solving:**
    Companies today are largely indifferent about which Linux distribution to use, with choices ranging from Ubuntu to NixOS to PopOS. This lack of a dominant opinion creates fragmentation and complexity. WendyOS simplifies this decision-making by providing a tightly integrated, optimized system purpose-built for edge development.

    **Why a Dedicated OS Matters:**
    - **Out-of-the-box readiness**: WendyOS includes essential libraries and tools for edge use cases, such as `lldb-server` for remote debugging, DNS-SD for service discovery, SQLite3 for local database management, and Swiftly for managing Swift toolchains
    - **Opinionated environment**: Reduces compatibility issues and ensures superior support for our software stack
    - **Unparalleled performance**: By controlling the OS, we deliver performance, security, and reliability optimizations impossible with generic Linux distributions
    - **Perfect timing**: With the majority of robotics, IoT, and edge projects in 2025 being greenfield, the market is positioned for a purpose-built OS

    **Flexible but Opinionated:**
    While WendyOS is the ideal foundation for our stack, the Wendy framework—including the debugger, deployer, developer tools, and frameworks—works seamlessly on other major Linux distributions. This respects existing investments while strongly encouraging adoption of WendyOS for optimal performance and integration.

    **Inspiration from Success Stories:**
    The success of SteamOS for Valve's Steam Deck, built by a small team, demonstrates the viability of creating a Linux variant tailored to a specific audience and use case. Similarly, we're building WendyOS to define the standard for edge computing.

    **More Than Just an OS:**
    Wendy is an entire ecosystem. By providing a unified platform, we enable an explosion of innovation in robotics, IoT, and edge AI. We're consolidating and simplifying the edge development experience, offering developers—both at companies and hobbyists—a clear path to easy development and production deployment while greatly reducing fragmentation.

    WendyOS positions Wendy as the standard for edge computing in the years to come, paving the way to be an "iOS" for all edge projects.
  </Accordion>

  <Accordion title="Wasn't this called Edge or EdgeOS?">
    Yes, the project was originally called Edge and EdgeOS. However, we ran into a fundamental problem: it was too hard to get the trademark for a generic word like "Edge."

    **Why we changed to Wendy:**
    We tend to name our products after video game characters, and Wendy is named after the boss in Aerie from Nier Replicant. It was such a strange name that people never forgot it—and that uniqueness became a strength.

    **The bigger realization:**
    We loved "Edge," but after 15 years, nobody understood what "edge computing" was or why they needed it. But a **robot or physical AI operating system**? That everyone understood.

    Every kid grows up wishing we'd have robots—from the Jetsons to C3PO and R2D2. You'll remember that more than some corporate jargon of "edge computing."

    **Making matters worse**: Cloud services started using terminology like "At the edge," "Edge Functions," and "Edge Caches," further muddying what "edge" even meant. The term became so overloaded that it lost all meaning.

    The name change wasn't just about trademarks—it was about clarity. Wendy immediately communicates what we're building in a way that "Edge" never could. The memorable name helps, but the clear positioning as an OS for robots and physical AI is what really resonates with developers and companies.
  </Accordion>

  <Accordion title="What is WendyOS?">
    WendyOS is a lightweight operating system designed for edge computing devices like NVIDIA Jetson and Raspberry Pi. It provides a simple, containerized platform for deploying applications written in Swift, Python, Rust, C++, and other languages.

    **Technical Details:**
    - **Base**: Debian-based Linux distribution
    - **Architecture**: Targets ARM64 architecture
    - **Purpose-built**: Optimized for robotics, IoT, and physical AI applications
  </Accordion>

  <Accordion title="Which language should I use for physical AI projects?">
    For physical AI projects, especially those involving moving parts or actuators, we **highly recommend** languages like Swift, Rust, and C++. Here's why:

    **RAM is precious on edge devices.** Physical AI applications run on devices with limited memory (typically 4GB to 64GB), and every megabyte counts when you're running AI models, sensor processing, and control systems simultaneously.

    **Memory management matters for safety.** Languages like Node.js (TypeScript/JavaScript) and Python use garbage collection, which can cause unpredictable memory spikes and latency. When you're controlling motors, servos, or robotic arms, these unpredictable pauses can be dangerous. You need deterministic performance.

    **Why Swift, Rust, and C++ are better for physical AI:**

    - **Predictable memory usage**: Swift uses Automatic Reference Counting (ARC), Rust uses ownership, and C++ gives you manual control. All three provide deterministic memory management without garbage collection pauses.
    - **Low memory overhead**: These languages have minimal runtime overhead compared to interpreted or JIT-compiled languages.
    - **Real-time capable**: No garbage collection pauses means consistent, predictable timing for control loops.
    - **Better for safety-critical systems**: When parts are moving, you cannot afford unpredictable behavior caused by GC pauses.

    **When TypeScript/JavaScript and Python are acceptable:**

    These languages are perfectly fine for:
    - Web servers and APIs on your device
    - Data logging and monitoring dashboards
    - High-level orchestration that doesn't control hardware directly
    - Prototyping and experimentation (but consider rewriting in Swift/Rust/C++ for production)

    **The bottom line:** If your code directly controls motors, actuators, or any moving parts, use Swift, Rust, or C++. If you're building web interfaces, APIs, or data processing pipelines that don't interact with physical hardware, TypeScript/JavaScript and Python are fine choices.

    Think of it this way: you wouldn't want a garbage collector deciding to pause your robot's control loop while its arm is mid-motion. Use the right tool for the job, and for physical AI, that means languages with predictable, deterministic memory management.
  </Accordion>

  <Accordion title="Why did you choose Swift as the core language for WendyOS?">
    We chose Swift as WendyOS's flagship language to solve a fundamental problem in edge development: **fragmentation**.

    Modern edge development is incredibly fragmented. Each company reinvents the wheel with overly complicated stacks, using multiple programming languages for tasks that should be straightforward. This creates unnecessary complexity and increases the barrier to entry for developers who must learn multiple languages and tools just to get started.

    **Swift unifies this fragmented ecosystem:**
    - **Single language for everything**: Eliminates the common but inefficient dual-language workflow (prototype in Python, rewrite in C++ for production)
    - **Memory safe and performant**: Uses Automatic Reference Counting (ARC), not garbage collection, making it deterministic and suitable for real-time systems
    - **Modern and readable**: Production-ready from the start, no need to rewrite code
    - **Excellent C/C++ interoperability**: Swift's "escape hatch" allows seamless integration with existing system libraries
    - **Community building**: One language means unified documentation, blog posts, libraries, and packages instead of fragmented solutions

    **Why not other languages?**
    - **Rust**: Excellent language but slow compile times, cumbersome syntax, and painful FFI work with C/C++ headers
    - **C++**: While extremely powerful, it can be complex to get something simple going. The common workflow of prototyping in Python and rewriting in C++ for production reveals that if C++ was truly productive for rapid development, companies would start with it, not end with it
    - **Python**: Great for prototyping but too slow for production, non-deterministic garbage collection, and no explicit type system. Industry consensus: "garbage collected languages are dead on arrival" for controlling actuators or dangerous equipment
    - **Zig**: Too early and unstable; not yet production-ready for mission-critical systems
    - **Java/Kotlin**: Require garbage collection and poor C interoperability story

    We looked to mobile development's success: Apple's decision to unify development under Swift is one of their greatest contributions to software development. WendyOS brings this same unification to edge computing, creating a common ecosystem where Swift becomes the lingua franca—just as JavaScript is for web developers and Kotlin is for Android developers.

    **Note**: We do support Python, Rust, C++, and other languages through containers and containerd (built with Docker or Apple Container). However, for the most reliable experience with our frameworks, we highly recommend Swift.
  </Accordion>

### Installation & Setup

<Accordions type="single">
  <Accordion title="Do I need Docker to build apps?">
    Not necessarily. Wendy uses containers for app builds and deployment, but Docker is only one of the supported builders.

    On an Apple Silicon Mac, Wendy tries [Apple Container](https://github.com/apple/container) first for `Dockerfile` and `Containerfile` builds when `--builder` is omitted, and falls back to Docker if Apple Container is unavailable or the build fails. Use `--builder apple-container` to require Apple Container, or `--builder docker` to force Docker.

    Docker is still required for Docker Compose projects (`docker-compose.yml` / `compose.yml`) on local provider runs. The Wendy CLI handles the local builder setup when it is needed.
  </Accordion>

  <Accordion title="How long does it take to install WendyOS on a device?">
    Installation time varies by device and internet speed:
    - **Download WendyOS image**: 5-15 minutes (depends on internet speed)
    - **Flash to storage**: 5-10 minutes
    - **First boot**: 2-3 minutes

    Total time: approximately 15-30 minutes.

    `wendy install` also lets you pre-configure the device so it's ready the moment it powers on: connect to WiFi on first boot, automatically install any important OS updates on first boot, and pre-enroll it so it securely links to Wendy Cloud on first boot (it links to your developer machine once you connect it over USB-C). Each of these is optional and can also be set up after the device boots.
  </Accordion>

  <Accordion title="Can I deploy Wendy apps to macOS or other Linux environments?">
    **Yes.** [Wendy for Mac](/docs/installation/wendy-agent-macos) is in beta for Apple Silicon Mac targets, and `wendy-agent` already works on Linux environments.

    We architected Wendy Agent (the program that manages apps) to be platform agnostic. This means you can deploy Wendy applications to:
    - **macOS** (Apple Silicon Mac targets in beta)
    - **Linux machines** of all kinds (x86_64, ARM64, etc.)
    - **Various Linux distributions** (Ubuntu, Fedora, Arch, etc.)
    - **Cloud servers** and development machines

    The Wendy Agent's platform-agnostic design allows it to manage applications across different operating systems and architectures, not just WendyOS devices. This flexibility means you can develop locally and deploy to WendyOS-powered edge devices, Linux boxes, and Apple Silicon Macs with the same CLI workflow.
  </Accordion>

  <Accordion title="Why do you recommend MLX over PyTorch or other AI frameworks?">
    We recommend **MLX** because NVIDIA Jetson devices (Orin Nano, AGX, Thor) use **unified memory architecture**—just like Apple M-series Silicon. This eliminates the complexity of managing separate CPU and GPU memory.

    **MLX is much simpler** because it takes advantage of unified memory without dealing with CPU/GPU memory transfers. This makes code cleaner, more maintainable, and easier to debug.

    **Compare the verbosity:**

    **MLX (Simple & Clean):**
    ```python
    import mlx.core as mx
    import mlx.nn as nn

    # Create model - automatically uses unified memory
    model = nn.Linear(784, 10)
    x = mx.random.normal((32, 784))

    # Forward pass - no device management needed
    output = model(x)
    loss = mx.mean((output - target) ** 2)
    ```

    **PyTorch (More Verbose):**
    ```python
    import torch
    import torch.nn as nn

    # Check device availability
    device = torch.device('cuda' if torch.cuda.is_available() else 'cpu')

    # Create model and move to device
    model = nn.Linear(784, 10).to(device)
    x = torch.randn(32, 784).to(device)  # Move data to device
    target = torch.randn(32, 10).to(device)  # Move target to device

    # Forward pass
    output = model(x)
    loss = torch.mean((output - target) ** 2)
    ```

    With MLX, you don't need:
    - Device checking (`cuda.is_available()`)
    - Manual tensor transfers (`.to(device)`)
    - Memory management between CPU and GPU
    - Worrying about data on wrong devices

    This simplicity is perfect for edge AI development where you want to focus on your models, not memory management.
  </Accordion>

  <Accordion title="Wait, MLX can run on NVIDIA?">
    **Yes!** As of [v0.29.0](https://github.com/ml-explore/mlx/releases/tag/v0.29.0), MLX supports CUDA, which is used on NVIDIA Jetson cards.

    See the [official MLX CUDA installation guide](https://ml-explore.github.io/mlx/build/html/install.html#cuda) for setup instructions.

    This is a game-changer for edge AI development:
    - **Test on your MacBook**: Develop and test MLX code on Apple Silicon
    - **Deploy to Jetson**: Run the exact same code on NVIDIA Jetson devices **without changing anything**
    - **No rewrites needed**: Your MLX code is truly portable across unified memory architectures

    The unified memory approach works seamlessly on both platforms, making MLX the ideal choice for developers working across Apple Silicon development machines and NVIDIA Jetson edge devices.

    You get the best of both worlds: Apple's MLX simplicity with NVIDIA's edge AI performance.
  </Accordion>

### Troubleshooting

<Accordions type="single">
  <Accordion title="My device isn't showing up in wendy devices list">
    Try these troubleshooting steps:
    1. Ensure your device is powered on
    2. Check that the USB cable is properly connected
    3. Verify the USB cable supports data transfer (not just charging)
    4. On macOS, grant permission when prompted to use the USB device
    5. Try a different USB port on your computer
  </Accordion>

  <Accordion title="The installation failed. What should I do?">
    If installation fails:
    1. Check that you have sufficient disk space on your storage device
    2. Verify your internet connection is stable
    3. Try a different storage device (SD card or NVMe SSD)
    4. Check the error message for specific details
    5. Ensure your storage device is properly formatted
  </Accordion>
